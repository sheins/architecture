Introduction to Front-End Architecture

1. What is Front-End Architecture?

  • Where files & assets go (how they’re organized)
  • Good architecture is a fundamental step in building a house. The same applies to applications/websites. Just as the structure for a house varies from the structure of a baseball stadium, different applications/websites can require different structure.

2. Mostly Conceptual

While the basic pieces/ideas are the same, architecture may vary based on:

  • Individual/Team Preferences
    • Directory naming `imgs` vs. `images`
    • Organized by section of site or type of asset? Probably both, but in what order…`app/about/css` vs. `app/css/about
  • Project Requirements
    • Responsiveness - Supporting mobile & desktop? Other devices?
  • Tools / Build Process
    • Preprocessors (like Sass/Less for CSS?) - Use `scss` folder instead of `css`
    • Frameworks - You’ll want a file structure to complement that framework - i.e. Django has its own style guide to follow

3. Modularity

  • Single Responsibility Principle
    • Certain sections of code should only be in charge of a single functionality
    • Resuseable Code
  • Sounds great in theory, but how does this apply to real life?
    • Fewer bugs by design
    • When there are bugs, they're easier to find, and to fix
    • End result? Happy programmers and happy clients
- Single responsibility principle - a term that is used in all aspects of code, from changing colors of links in CSS to making a footer that appears on every page. - You have a website, and the navigation bar appears on every page. You're not going to rebuild that navigation bar at the top of every HTML file you create, you're going to make a module/component file, called something like nav.html, and then tell your code to include that on every page. - Why? A) copying and pasting that much will give you carpel tunnel. B) Hard to maintain - if the client wants to change what links are in their nav bar, you'd have to edit every HTML file with the navigation. Things change in programming all the time in ways you wouldn't expect, this makes it easier to make those changes in the future -Frustrating to work on "legacy code" that wasn't well thought out. Hard to locate bugs and squash them entirely without modularity. If you jump into a project and the nav bar is set up where you have to edit it on every page, and you miss a page, there's a bug floating around.

The Structure

App Directory

  • Index.html
  • Assets
  • Dist

Index.html

  • The most important page for your site - the Home Page
  • What gets served (without /index.html appended to the end of the url).
  • When you view www.amazon.com, you're viewing amazon.com's index.html file, for example.

Assets

  • Images (.png/.jpg files)
  • Styles (.css files)
  • Scripts (.js files)
  • Fonts (.otf, .ttf files)
  • Views (partial .html files)

Dist

  • Short for "distribution"
  • Production-ready
  • Files are made to load quickly - they're minified
    • Removes spaces, replaces human-created function names to shorter names (like getDrink() -> x())
    • Not very human readable, but are read quickly by the browser for better performance
  • Can be manually created, but usually developers use a tool like Gulp to automate the process.

In dist, you have...

  • `index.html` (`dist/index.html`)
  • Assets
  • Views

Other things important for good architecture

  • Documentation
  • Naming Conventions
  • Build tools
Documentation - Helps developers who are new to the project hit the ground running. What's the project about? What are the naming conventions? Naming conventions - will you name your pages "blog.html" or "blog-page.html"? Things seem insignificant when you first start a project, but once you have a 30 different folders that are all named differently a year down the road, it will cause a lot of confusion Build tools - Help simplify and automate the process.

Setting up your project wearing you out?

Yeoman is here to help!

What is Yeoman?

  • Tool that sets up your project's file stucture
  • Can be used with a variety of languages using generators to create scaffolding
  • Install via NPM 
    npm install -g yo
  • Give commands to Yeoman using Yo in the CLI
- Yeoman is like a personal assistant. It'll do the boring tasks such as setting up folders and preprocessors for you. But like any good assistant, it'll need exact information in order to properly complete it's task. That's where generators come in. - Generators are different project structures, called scaffolding, in a variety of lanuages created by other developers. There are currently over 4,000 different generators in the Yeoman to choose from. You can even create your own! - Yeoman is an npm install, which means that node and npm need to be downloaded on your computer in order to use Yeoman. - We talked about npm last month, but if you missed it, you can review the presentation at the coding and cocktails website. - Start working with Yeoman using the yo command in your command line.

Setup Yeoman project using Fountain Webapp Generator

- Since we are going to be creating a web app, we'll be making our new project using the generator called Fountain Webapp. Each generator comes with it's own features depending on the langauage. We'll talk about Fountain Webapp's features in the next slider.

Fountain Webapp Features

  • Creates file structure for project
  • CSS autoprefixing
  • Compiles SASS
  • Lints scripts
  • Live Reload of project
  • Image Optimization
  • And much more!
- Our generator does a lot of cool things for us. As you go through your new project, you'll see that the file structure Sara talked about has been created for us. - Also, if you are using cool CSS3 functionalities like transitions and animations, webapp will automatically add the prefixes needed to make them work with all browsers. - It will also compile your SASS file. SASS is a CSS preprocessor that allows developers to use variables and logic in their css. By compiling your SASS automatically, it makes those css variables and logic readable to the browser. - Javascript not working? Webapp will lint your script, showing you the exact line where the problem is occuring and providing tips on best practices. - As part of our project, a tool called Gulp has been installed. By running the gulp server, it will automatically reload your project each time you make a change to the css. You'll no longer have to press the reload button to see your changes. Gulp also helps with image optimization and css minifying, which will help your webite load super fast and create a great experience for visitors. - There are many other things that Gulp can do so feel free to check out the website to learn more!

Helpful commands

  • To turn on server and view project
    • npm run serve
  • Compiles and optomizes files to make production ready
    • npm run build
  • Preview production files
    • npm run serve:dist
  • For help using Yeoman
    • yo help
  • For help using generator
    • yo [generator name] --help
  • Running into errors with yo commands? 
    • yo doctor
- To see the project you are working on, you'll need to run npm run serve. This is a server that will be running in the background while you are making changes to the project and updating the files that you are changing accordingly. - Once you are ready to push your new site live, the npm run build command will use the css you've added and minify it into a smaller file that is difficult to humans to read but allows the browser to run the file faster. - Sometimes things go wrong during the minification process, which is why you should run the npm run serve:dist command. This command will show you how your site will look when it is live using the minified files. - If you need help with Yeoman or the generator you can run the command yo help or yo and the name of your generator. - If you are using yeoman and running into errors you don't understand, you can run yo doctor and that command will troubleshoot issues for you and give you recommendations on how to solve it, such as updating npm or updating yeoman.
